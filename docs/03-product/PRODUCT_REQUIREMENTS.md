# PRODUCT REQUIREMENTS

## Purpose

Данный документ содержит полный каталог функциональных (FR) и нефункциональных (NFR) требований SHIK Platform.

Каждое требование имеет:

- уникальный идентификатор;
- описание;
- бизнес-ценность;
- приоритет;
- критерии приемки (Acceptance Criteria);
- связи с RTM.

---

# Priority Model

Используется модель MoSCoW.

| Priority | Description |

| Must | Обязательно для MVP |
| Should | Желательно |
| Could | Может быть реализовано позже |

## FR-013 Menu Management

### Description

Платформа должна позволять уполномоченным сотрудникам управлять меню через Back Office без изменения исходного кода и без публикации новой версии приложений.

### Priority

Must

### Business Value

Обеспечить оперативное изменение ассортимента, цен, доступности и отображения блюд во всех каналах продаж.

### Actors

- Owner
- Restaurant Manager
- Marketing Manager
- Menu Administrator

### Functional Scope

- создание и редактирование меню;
- первоначальный импорт из проверенного JSON или CSV;
- публикация и снятие меню с публикации;
- Draft, Preview и Published lifecycle меню;
- подготовка полного каталога до публикации клиентских приложений;
- настройка меню для отдельных филиалов;
- настройка времени действия меню;
- управление порядком категорий и блюд;
- предварительный просмотр перед публикацией;
- планирование публикации изменений;
- история версий меню.

### Acceptance Criteria

- уполномоченный пользователь может создать, изменить и опубликовать меню;
- полный каталог можно подготовить и проверить до публикации Customer App в App Store и Google Play;
- импорт отклоняет дублирующиеся идентификаторы и строки без обязательных полей;
- изменения отображаются в Customer App, POS, QR-меню и других подключенных каналах;
- клиентские приложения получают опубликованный каталог через Backend API;
- публикация не требует обновления приложения;
- меню может быть настроено для всей сети или отдельного филиала;
- неопубликованные изменения не видны клиентам, но доступны уполномоченному пользователю в Preview;
- порядок категорий и блюд одинаково воспроизводится во всех подключенных каналах;
- каждая публикация фиксируется в Audit Log;
- система позволяет восстановить предыдущую опубликованную версию меню;
- позиции, недоступные в конкретном филиале, не должны отображаться как доступные для заказа.

## FR-014 Product Management

### Description

Платформа должна позволять управлять карточками блюд, товаров, наборов и дополнительных позиций.

### Priority

Must

### Business Value

Обеспечить единый источник актуальной информации обо всех продаваемых позициях.

### Actors

- Owner
- Restaurant Manager
- Menu Administrator
- Marketing Manager

### Functional Scope

- стабильный идентификатор;
- название;
- описание;
- категория;
- фотографии;
- главная фотография и порядок галереи;
- загрузка, замена и безопасное удаление фотографий;
- цена;
- состав;
- аллергены;
- вес и объем;
- пищевая ценность;
- халяльная маркировка;
- Draft, Published и Hidden lifecycle карточки;
- статус доступности;
- Stop List отдельно по филиалам;
- ручные признаки Popular, New и Featured для MVP;
- модификаторы;
- размеры;
- варианты приготовления;
- рекомендации;
- сопутствующие товары;
- порядок отображения;
- филиальная доступность;
- Preview до публикации.

### Acceptance Criteria

- пользователь с соответствующим разрешением может создать и изменить карточку продукта;
- новая карточка создается в статусе Draft и не видна клиентам до публикации;
- обязательные поля проходят валидацию;
- цена не может быть отрицательной;
- изменение цены, описания или фотографии не требует выпуска новой версии приложения;
- продукт может иметь одну или несколько фотографий;
- фотография загружается через Backend API в Object Storage, а ссылка и метаданные сохраняются в PostgreSQL;
- пользователь может заменить фотографию, назначить главную и изменить порядок галереи;
- удаление фотографии разрешается только после проверки использования и выполняется с безопасной отложенной очисткой;
- Preview показывает карточку так, как она будет отображаться после публикации;
- Popular, New и Featured управляются вручную через Back Office в MVP;
- продукт может иметь модификаторы и варианты;
- халяльная маркировка отображается в подключенных клиентских интерфейсах;
- изменения карточки фиксируются в Audit Log;
- Hidden исключает продукт из каталога без удаления и не изменяет исторические заказы;
- Stop List управляет временной доступностью отдельно по филиалам и не заменяет Hidden;
- удаление продукта, использованного в заказах, не должно уничтожать исторические данные;
- система поддерживает разные цены и доступность по филиалам;
- повторный импорт обновляет данные по стабильному идентификатору и не создаёт дубликаты.

## FR-016 Kitchen Display System

### Description

Платформа должна предоставлять Kitchen Display System для обработки заказов на кухне в режиме реального времени.

### Priority

Must

### Business Value

Сократить время приготовления, уменьшить количество потерянных заказов и повысить прозрачность работы кухни.

### Actors

- Kitchen Staff
- Senior Cook
- Kitchen Administrator
- Restaurant Manager

### Functional Scope

- получение новых заказов;
- распределение по кухонным станциям;
- изменение статусов;
- таймеры;
- отображение комментариев;
- отображение аллергенов;
- частичная готовность заказа;
- техкарты;
- стоп-лист;
- предупреждения о задержках;
- звуковые и визуальные уведомления;
- работа при кратковременной потере сети.

### Acceptance Criteria

- подтвержденный заказ появляется на кухне не позднее чем через 3 секунды после серверного события при нормальном соединении;
- заказ содержит состав, количество, модификаторы и комментарии;
- сотрудники могут менять только разрешенные статусы;
- критические аллергены и особые пожелания визуально выделяются;
- система показывает время ожидания и время приготовления;
- готовность отдельных позиций может отмечаться независимо;
- изменения синхронизируются с клиентским приложением и POS;
- повторная доставка одного события не создает дубликат заказа;
- действия сотрудников фиксируются;
- при кратковременной потере связи экран сохраняет уже полученные активные заказы и синхронизируется после восстановления соединения.

## FR-017 POS Operations

### Description

Платформа должна предоставлять кассовый интерфейс для создания заказов, приема оплаты и управления сменой.

### Priority

Must

### Business Value

Объединить заказы на кассе с мобильным приложением, кухней, складом и аналитикой.

### Actors

- Cashier
- Restaurant Manager
- Financial Manager

### Functional Scope

- создание заказа;
- поиск и добавление товаров;
- модификаторы;
- скидки и промокоды;
- привязка клиента;
- прием оплаты;
- печать и отправка чека;
- возвраты;
- отмены;
- открытие и закрытие смены;
- внесение и изъятие наличных;
- синхронизация с Kitchen Display System;
- работа с фискальным оборудованием через интеграционный слой.

### Acceptance Criteria

- кассир может создать заказ за минимальное количество действий;
- недоступные позиции нельзя добавить без специального разрешения;
- итоговая сумма рассчитывается автоматически;
- каждая оплата связана с заказом и сменой;
- возврат требует разрешения и указания причины;
- заказ после подтверждения передается на кухню;
- кассир видит статус приготовления;
- все финансовые операции фиксируются в Audit Log;
- POS не хранит платежные секреты в открытом виде;
- при временном отсутствии связи применяются утвержденные правила offline-работы и последующей синхронизации;
- конфликт синхронизации не должен приводить к двойной оплате или двойному заказу.

## FR-031 Legal Documents, Public Offer and Personal Data Consent

### Description

Платформа должна обеспечивать юридически проверяемые пользовательские сценарии для публичной оферты, политики обработки персональных данных и отдельных согласий.

### Priority

Must

### Functional Scope

- публичная оферта с версией, датой вступления в силу и реквизитами продавца;
- политика обработки персональных данных;
- отдельное согласие на обработку персональных данных;
- отдельное необязательное согласие на маркетинговые коммуникации;
- отображение действующих документов до юридически значимого действия;
- фиксация версии, хеша, времени, канала и технических доказательств принятия;
- привязка принятой версии оферты к заказу;
- отзыв маркетингового согласия;
- запрос на доступ, исправление, экспорт и удаление данных;
- Legal & Privacy Center в Customer App;
- управление Draft, Review, Published и Superseded версиями документов в Back Office;
- неизменяемая история опубликованных версий и действий администратора.

### Acceptance Criteria

- согласие на обработку персональных данных оформляется отдельно от оферты и других документов;
- маркетинговое согласие выключено по умолчанию и не блокирует оформление заказа;
- заказ невозможно создать без действующей версии оферты и подтверждаемого факта её принятия;
- сервер проверяет принятую версию и сохраняет неизменяемый snapshot юридических данных заказа;
- опубликованный документ, использованный в принятии или заказе, нельзя удалить или изменить задним числом;
- пользователь может просмотреть действующие документы и управлять разрешёнными согласиями;
- персональные данные и доказательства согласий хранятся в российском production-контуре;
- внешний провайдер персональных данных не используется без privacy, legal and architecture review;
- production launch блокируется до юридического утверждения текстов и выполнения release gates CMP-1908.
